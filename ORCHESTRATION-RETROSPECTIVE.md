# Ретроспектива оркестрации — прогон #1 (блок «Автоматизация/Бот»)

Дата: 2026-06-21. Задача: спроектировать архитектуру + план блока «Автоматизация» (без кода).
Это **первый боевой прогон** связки VibeProxy + Codex + Workflow в этом репозитории. Цель файла —
зафиксировать, как отработали каналы/модели как агенты, что не состыковалось, и что улучшить в
следующий раз. Артефакты прогона: [AUTOMATION-ARCHITECTURE.md](AUTOMATION-ARCHITECTURE.md),
[AUTOMATION-PLAN.md](AUTOMATION-PLAN.md).

## 1. Что запускалось (топология прогона)

Четыре канала параллельно, с намеренно **перекрывающимися** углами (для перекрёстной верификации и
сравнения моделей):

| Канал | Модель | Роль | Режим |
|-------|--------|------|-------|
| **Workflow** (in-house) | Opus 4.8 ×9 агентов | Основной конвейер: понять код (3) → дизайн 5 слоёв → критика (1) | background, structured output |
| **Codex** (MCP) | gpt-5.x, priority/high | Фезибилити instagrapi-фильтров + адверсариальная критика очередей/лимитов | foreground, read-only sandbox, реальный репозиторий |
| **VibeProxy** (MCP) | Kiro-Opus 4.8 | Независимый дизайн движка действий | foreground |
| **VibeProxy** (MCP) | Kiro-Sonnet 4.6 | Независимый план FSD-слайсов + UX фильтров | foreground |

Метрики Workflow: **9 агентов, ~689k токенов субагентов, 143 tool-вызова, ~12.8 мин** (766 с).
Заземление до запуска: 6 параллельных чтений (карта 3 слоёв) в main-контексте.

## 2. Как отработали модели (оценка по факту)

### Workflow / in-house Opus — ★★★★★ несущая ценность
- 3 understand-агента (Explore) дали точные точки переиспользования с реальными классами/методами —
  на них легла вся карта «что не писать заново».
- 5 design-агентов выдали детальные спеки (9 таблиц с полями, 2 Python-endpoint'а, контракт плагина).
- **Адверсариальный критик — лучший агент прогона.** Нашёл то, что внешние каналы НЕ увидели:
  - **блокер `auth()->id()` внутри `InstagramClientService`** (в очереди вернёт `null→0`, ломает
    ownership и лимиты) — не заметил никто, кроме критика;
  - **отсутствие scheduler-контейнера** (оба backend-дизайна строят на `schedule()->everyMinute()`,
    которому негде крутиться);
  - **конфликт драйвера очереди** (config=database vs worker=redis);
  - все межслойные противоречия свёл в список с разрешением.

### Codex — ★★★★★ лучший по «реальный репозиторий + внешний API»
- Дал самые конкретные числа стоимости: запросов на цель по каждому фильтру, оценка «600–1000
  IG-запросов на 100 целей», потолки N — всё с привязкой к `file.py:line`.
- Топ-5 рисков (взрыв запросов, неатомарные лимиты, дубли комментов, backlog burst, in-memory lock)
  совпали с критиком — **независимое подтверждение** ключевых решений (БД = источник истины).
- Вывод: внешне-API-фезибилити и «прочитай реальный код и посчитай» — профильный канал Codex.

### VibeProxy Kiro-Opus — ★★★★☆ сильное независимое второе мнение
- Полноценный самостоятельный дизайн движка: контракты, DTO, статусные машины, таблицы, поток,
  точки расширения. ~6050 выходных токенов, связно и по делу.
- **Конвергенция с in-house** по ключевым идеям (плагин-контракт действий, «два режима = один
  движок», материализация под-действий) → повышает уверенность, что архитектура верная.
- **Дивергенция**: предложил delay()-Job на под-действие и счётчик в Redis — оба отвергнуты в пользу
  БД-driven (см. §3). Это нормально для независимого мнения, но требует сведения.

### VibeProxy Kiro-Sonnet — ★★★☆☆ хороший результат, упал транзиентно с первого раза
- **Первый вызов: HTTP 500** — `event stream fatal: failed to read prelude: context deadline
  exceeded`. Первоначально списано на per-account spacing, но **контрольные тесты опровергли это**
  (см. ниже) — фактически это **транзиентный upstream read-timeout** на длинной генерации (~4442 out).
- **Повтор тем же промптом — успех.** Дал лучший конкретный UX: ASCII-раскладку формы фильтров,
  4 группы, `FilterBadge`, sticky footer, drag-select сетку часов. Выход напрямую лёг в §8.3 архитектуры.
- **Проверка параллелизма (контрольные тесты после прогона):** одновременные `ask` Opus+Sonnet прошли
  ОБА — и на тривиальных, и на тяжёлых (out≈1.7k/2.1k) промптах. Из `vibeproxy-server/CLAUDE.md`:
  session-affinity ключ = `hash(model+system+first_msg)` → РАЗНЫЕ модели идут параллельно даже на
  одном аккаунте; сериализуются лишь идентичные (same-key) запросы (per-auth spacing 2–15с).
  **Вывод исправлен:** «нельзя параллелить Opus+Sonnet» — неверно. Падение было транзиентом, лечится
  ретраем; для длинных генераций — поднять client read-timeout.

## 3. Что НЕ состыковалось (конфликты и их разрешение)

Перекрытие углов было намеренным — оно вскрыло реальные развилки. Каждую свёл оркестратор:

| Конфликт | Кто за что | Разрешение | Обоснование |
|----------|-----------|-----------|-------------|
| Распределение во времени | Kiro-Opus + engine-агент: delay()-Job · Codex + DB-агент + критик: БД-строки+диспетчер | **БД-driven** | delay()-Job → catch-up storm после простоя, неотменяемы |
| Учёт лимитов | engine-агент: COUNT по логам · DB-агент + Codex + критик: счётчик + FOR UPDATE | **Счётчик-таблица** | COUNT по логам: гонки, логи чистятся, нет атомарного резерва |
| Имена таблиц/статусов | два разных набора имён у двух backend-агентов | **Один канонический набор** | иначе несовместимые миграции |
| Учёт userId в очереди | не заметил никто, кроме критика | **auth-context-free клиент** | блокер исполнения из очереди |

**Главный системный вывод:** при fan-out двух агентов на ПЕРЕКРЫВАЮЩУЮСЯ зону (здесь — оба
проектировали backend-движок) расхождение по несущим решениям неизбежно и требует обязательной фазы
**критика + синтез оркестратором**. Перекрытие дало пользу (адверсариальность вскрыла развилки), но
оплачено токенами на дублирующий дизайн.

## 4. Что улучшить в следующей оркестрации

1. **Pre-flight «shared decisions» ДО fan-out.** Перед параллельным дизайном backend зафиксировать
   канонические имена таблиц, механизм лимитов и модель времени одним коротким агентом/решением.
   Это срезало бы ~половину работы по сведению конфликтов §3. (В план уже вынесен чек-лист
   «фиксируем ДО кода».)
2. **Острее партиционировать fan-out.** Два backend-агента (`laravel-engine` + `automation-engine`)
   сильно перекрылись. Либо один агент на backend-оркестрацию, либо перекрытие делать ЯВНО
   адверсариальным (один проектирует, второй атакует) — а не два независимых дизайна одного и того же.
3. **Параллелить РАЗНЫЕ модели Kiro можно** (Opus+Sonnet — разные affinity-ключи, идут параллельно).
   Сериализовать нужно лишь идентичные (same-key) вызовы. Транзиентные `failed to read prelude`
   таймауты на длинных генерациях лечить idempotent-ретраем `ask`; при стабильной проблеме — поднять
   client read-timeout. Не строить ложных ограничений на параллелизм по моделям.
4. **Маршрутизация по сильным сторонам каналов** (подтверждено прогоном):
   - внешне-API-фезибилити + «посчитай по реальному коду» → **Codex** (priority/high);
   - адверсариальная критика архитектуры → **in-house критик** (поймал блокеры);
   - независимое второе мнение по дизайну → **Kiro-Opus** (конвергенция = уверенность);
   - конкретный UX/верстка/раскладка → **Kiro-Sonnet** (но в одиночку).
5. **Сохранить обязательную фазу Critique.** Она окупилась: три блокера и все межслойные конфликты
   найдены именно ей. Без неё прогон выдал бы красивый, но нерабочий дизайн.
6. **Structured output (schema) в Workflow** сильно упростил синтез — оставить как дефолт для
   fan-out, где результаты нужно сводить.

## 5. Тех-заметки по каналам (для следующего раза)

- **Codex config** по правилу глобального `CLAUDE.md`: глубокий запрос → `service_tier: priority` +
  `model_reasoning_effort: high`. Сработало штатно, read-only sandbox + реальный cwd — корректный
  разбор с `file:line`.
- **VibeProxy `ask`**: Kiro-Opus ~2.9k вход / ~6k выход — норм. Разные модели параллелятся (session-
  affinity по `hash(model+system+first_msg)`). Падение Sonnet — транзиентный read-timeout на длинной
  генерации, не конкуренция и не размер промпта (повтор тем же промптом прошёл). Kiro-аккаунтов 2.
- **Workflow**: 9 агентов / 689k токенов / 12.8 мин на задачу проектирования такого масштаба —
  адекватно; цена в основном на 5 design-агентах. Перекрытие двух backend-агентов — главный резерв
  экономии.
- **Заземление перед fan-out** (6 чтений карты слоёв в main-контексте) себя оправдало: промпты
  агентам ссылались на реальные файлы/классы, агенты не плутали.

## 5ter. Прогон #1c — Wave 2/3 (логика + фронт, агенты пишут код)

Полная сборка MVP (комментарии хэштег/гео, полу-ручной) силами агентов с интеграцией оркестратором.

**Ключевой инцидент — конкурентный `delegate_edit` падает.** Запуск ДВУХ Kiro `delegate_edit` разом
(Opus=движок + Sonnet=парсинг) → ОБА не дали результата (Opus: `failed to read prelude: context
deadline exceeded`, Sonnet: пусто; backend-laravel не тронут). Вывод уточнён против §2: одиночный
`ask` разных моделей параллелится, но **многошаговый `delegate_edit` (8+ запросов на агента) к этому
не готов** — сустейнед-объём запросов + per-auth spacing рвут поток. Лечение: `delegate_edit`
сериализовать ИЛИ разводить по РАЗНЫМ каналам (Codex/Kiro/Claude), не по разным моделям одного Kiro.

**Восстановление сработало:** Codex(движок, backend-laravel) ‖ Claude-Agent(фронт, frontend-vue) —
разные каналы + разные директории → оба успешно конкурентно. Затем Kiro-Opus(парсинг) одиночно → ок.

| Юнит | Канал | Итог |
|------|-------|------|
| Движок (15 файлов) | Codex fast | ✅ php -l чисто; корректный `FOR UPDATE`, CAS, backoff, reconcile |
| Фронт-MVP (5 ui + 4 entity + 8 feature + 5 widget + page) | Claude-Agent (157k tok, 123 tools, ~14м) | ✅ самопроверка eslint+vue-tsc чисто |
| Парсинг (клиент+фильтры+Job+контроллеры) | Kiro-Opus delegate_edit (40 шагов) | ✅, но **вышел за скоуп** (сам прописал routes+bind) |

**Уроки интеграции (за оркестратором):**
- **Замораживать ТОЧНЫЕ пути/методы API до fan-out, не только имена ресурсов.** Kiro дал
  `/automation/tasks/{id}/parse` + POST-корзину, фронт ждал `/automation/{id}/parse-targets` + PATCH —
  пришлось сводить руками. Контракт в §архитектуры был недостаточно директивен по путям.
- **Агенты выходят за объявленный скоуп** (Kiro правил routes/provider; settings-контроллер при этом
  пропустил). Always verify — что сделано лишнего и что НЕ сделано.
- **Швы за оркестратором** (роуты под контракт фронта, channel `automation-task`, schedule
  `automation:dispatch`, биндинги движка с тегом плагинов, миграция `media_thumbnail_url`,
  фронт-роут+таб, `AutomationSettingsController`) — ~10 правок, быстро и когерентно.

**Сквозная верификация (главная модель):** BE `php artisan test` 231✓; PY `pytest` 176✓; `vue-tsc`
чисто (1 предсущ. ошибка в quasar.config.ts); 14 маршрутов; миграции применены; контейнеры
`scheduler`/`automation-worker` живы; **UI вживую в Chrome** (логин→/automation→аккордеоны→табы, 0
ошибок консоли); **API-roundtrip** (POST создаёт task+parse_run, GET возвращает). Живой IG-парсинг/
коммент НЕ запускался (риск реального аккаунта — по решению владельца).

## 5quater. Прогон #1d — живой прогон реальным аккаунтом (через браузер + Chrome DevTools)

Цель: полностью проверить автоматизацию на живом IG-аккаунте (`desyatnikov_666`), 3 задачи.
**Результат: пройдено.** Реальный комментарий (`comment_pk 18106496032981248`) и реальный лайк
(`has_liked:true`) опубликованы; очереди/диспетчер/лимиты/cooldown/идемпотентность подтверждены.

**Найдено и пофикшено 4 реальных бага** (живой прогон вскрыл то, что тесты/типы не ловят):
1. **accountId не синкался в черновик парсинга** (frontend-дедлок): `draft.accountId` ставился только
   внутри клик-хендлера, а кнопка включалась по `canStartParse`, который требует `draft.accountId` →
   кнопка не включалась никогда. Фикс: реактивный `watch(accountId)` пишет в `draft.accountId`.
2. **Рассинхрон формата ответа `store`**: бэк возвращал `data:{task,parse_run}`, фронт ждал задачу
   плоско → `POST /automation/undefined/parse-targets` (500). Фикс: бэк возвращает task как `data`.
   (Прямое следствие «не заморозили точный контракт ответа» — урок из §5ter подтвердился на практике.)
3. **Broadcast payload too large роняет job**: `parse_*` activity-логи клали полный `instagram_response`
   в `response_summary` → Reverb «payload too large» → исключение синхронного broadcast валило
   `ParseTargetsJob`. Фикс: (а) `ActivityLoggerService` оборачивает broadcast в try/catch (мониторинг
   не должен ронять бизнес-операцию), (б) убран тяжёлый `instagram_response` из summary parse-методов.
4. **Review-таблица не авто-обновляется после async-парсинга**: UI зовёт `fetchTargets` сразу после
   dispatch, а job ещё не завершён → «Нет целей». Обойдено вручную; нужен правильный фикс (подписка на
   прогресс/поллинг до завершения парса) — вынесено в follow-up.

**Инфраструктурные грабли (не баги кода):**
- **`queue:work` кэширует код при старте** — после правок надо рестартить ВСЕ воркеры (queue-worker,
  automation-worker). Первый прогон упал, т.к. queue-worker был поднят 2ч назад (старый код без
  automation-биндингов: `Target [ParseRunRepositoryInterface] is not instantiable`).
- **LLM-провайдер не настроен** (`llm_settings` пуст, нет API-ключа) → авто-генерация текста коммента
  не работает. Исполнение коммента проверено инъекцией `comment_text` в payload (плагин использует
  готовый текст без LLM). Авто-генерация — это конфиг владельца, не баг.

**Что подтверждено живьём:** candidates+enrich по реальному IG (формат `feed/user` — TODO Codex закрыт),
фильтры, denорм-снимок цели (follower/following/avg-likes/last-post/thumbnail), корзина (PATCH exclude),
`ScheduleActionItemsJob`, диспетчер `automation:dispatch`, `RateLimitGuard` (резерв+откат на фейле),
CAS-claim, `commentMedia`/`addLike`, cooldown, счётчики задачи, прогресс. Движок **action-agnostic** —
comment и like через один конвейер, разные плагины.

**Делегирование в этом прогоне:** живой тест — интерактивный последовательный цикл на ОДНОМ аккаунте
(Python `account_lock` + антибан сериализуют IG-вызовы), параллелить нельзя; фиксы делались инлайн на
критическом пути (быстрее round-trip к агенту). Делегирование Kiro/Codex остаётся для Phase 2+ (фан-аут).

## 5quinquies. Прогон #2 — Phase 2 (full_auto), фан-аут 2 агента

Фан-аут по разным каналам/директориям (урок §5ter учтён): **Codex → backend** (full_auto
авто-планирование + идемпотентность) ‖ **Claude-Agent → frontend** (full_auto-поток + авто-refresh
review #4 + выбор действия comment/like). Непересекающиеся директории → конкурентно без конфликтов.

**Проверено живьём:**
- **full_auto end-to-end**: `СТАРТ (АВТО)` → задача → парсинг → бэк САМ диспатчит `ScheduleActionItemsJob`

**Баг #1b** (найден в этом прогоне): «Назад к настройке» (`resetBuilderHandler`) звал `resetDraft()`,

шла часами, реальное `now()` тоже ушло вперёд, `is_due=YES`. **Сверять с фактическим `now()` ДО вывода
о TZ-баге** (а не сравнивать с timestamps из ранних сообщений сессии).

## 6. Итоговая оценка прогона

Связка отработала по назначению: **перекрёстная верификация поймала то, что одиночная модель почти
наверняка пропустила бы** (блокер `auth()->id()`, отсутствие scheduler, конфликт очереди). Цена —
дублирующая работа на перекрытии backend-дизайна и один транзиентный таймаут Kiro-Sonnet (лечится
ретраем; параллелизм разных моделей подтверждён рабочим). Перекрытие устранимо pre-flight «shared
decisions».
Архитектура и план получены полные и сведённые. Готовность к Phase 0 — есть.
