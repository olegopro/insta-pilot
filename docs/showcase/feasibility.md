# Вердикт осуществимости — что реально в IG, а что только локально

> Проверено: офиц. дока **instagrapi 2.10.6** (Context7 `/subzeroid/instagrapi`) + инвентаризация репо
> (субагент Explore) + анализ оркестратора. Codex был недоступен (402 `deactivated_workspace`).
> Это фундамент архитектуры — менять только при доказанном опровержении на живом аккаунте (Phase 0).

## Крукс

**Произвольное перетаскивание сетки профиля с сохранением в аккаунт через приватный IG API — НЕВОЗМОЖНО.**
Сетка профиля в Instagram жёстко обратно-хронологическая; ни в API, ни в самом приложении нельзя задать
произвольный порядок постов. Единственный реальный рычаг порядка — **закрепить (pin) до 3 постов сверху**
(`media_pin` / `media_unpin`; для Reels `clip_pin` / `clip_unpin`).

Следствие для задумки: идея «обёртки над лентой» — правильная, ровно так работают индустриальные
планировщики (Later / Planoly). **Перетаскивание = локальная доска-планировщик** (порядок, пометки
«реклама», локальное скрытие — в нашей БД, в IG не уходят), а **реальные мутации до аккаунта** — это
edit подписи / archive / delete / pin.

## Таблица операций (источник — instagrapi docs)

| Операция | Метод instagrapi | Реально сохраняется в IG? | Фаза |

| Скрыть пост | `media_archive(media_id)` | ✓ (= «скрыть»: уходит из профиля в архив) | 3 |
| Показать пост обратно | `media_unarchive(media_id)` | ✓ | 3 |

## Что из задумки куда ложится

| Задумка пользователя | Реализация |
|---|---|
| Phone-simulator: профиль + сетка 3-в-ряд + аватар/подписчики/подписки | **Реально через IG** (`user_info` + `user_medias`), read-only. |
| Перетаскивать посты, менять порядок | **Локально** (overlay `board_position`) — доска-план, в IG не уходит. |
| Скрывать посты | Двойственно: **локально** (`is_hidden_local`, только в нашем UI) ИЛИ **реально** (`media_archive` — уходит из профиля). В UI развести явно. |
| Редактировать содержимое | **Реально** — `media_edit` (подпись/usertags/локация). Картинку/видео уже опубликованного поста менять нельзя (только cover у Reels — `clip_change_cover`). |
| Сохранить → появилось в аккаунте после refresh | **Реально** для edit/archive/delete/pin. Для произвольного порядка — нет (только pin). |
| Пометка «реклама», регистрация в локальной БД | **Локально** (`is_ad` + `is_tracked`) — полностью реализуемо. |
| Подтягивать официально-продвигаемые посты | **Нет** на личном аккаунте через приватный API. Только локальная пометка. (Бизнес/Graph API — отдельная тема, не трогаем.) |
| Аналитика по своим постам | Личный аккаунт: **counts-lite** (likes/comments/views из `user_medias` + динамика снимков). Полные insights — только бизнес-аккаунт (future). |
| Умный автопостинг | **Реально** (`*_upload`), но риск-зона (anti-ban) — отдельная опциональная фаза, не в фундаменте. |

## Нюансы, которые проверит Phase 0 (на живом аккаунте)

1. **Форма ответа `user_medias`.** `user_medias_paginated` отдаёт `Media`-модели (pydantic), а проектный
   `_serialize_media` (`helpers.py`) написан под raw-dict из `private_request`. Phase 0 фиксирует, какой
   путь даёт полные и чистые данные (caption, counts, thumbnail, media_type, resources карусели,
   **флаг pinned**) — `media.dict()`-адаптер ИЛИ `private_request "feed/user/{pk}/"` + `_serialize_media`
   (как в `_fetch_user_medias_raw`, у которого есть TODO про форму ответа).
2. **Как приватный API отдаёт флаг pinned** (закреплённые посты) — нужно для корректной сетки. Зафиксировать.
3. **`media_edit` на уже опубликованном посте** — реально ли применяется и виден ли после refresh; какие
   поля принимает (caption точно; usertags/location — проверить).
4. **`media_archive`/`unarchive`** — уходит ли пост из `user_medias` и появляется ли в `archive_medias`.
5. **`media_pin`/`unpin`** — как именно отражается закрепление в ответе профиля/медиа.
6. **`media_delete`** — необратим: НЕ гонять автоматически; проверять только на специально выделенном
   расходном посте по явному указанию владельца.
7. **Insights** — подтвердить, что на личном (не бизнес) аккаунте `insights_*` падают предсказуемой
   ошибкой (тогда Phase 5 для личного = counts-lite).
