# Ретроспектива и мониторинг оркестрации

Назначение файла — живой лог наблюдений за мультиагентной оркестрацией в этом репозитории:
производительность и сбои каналов, нюансы механизма делегирования, что улучшить в следующих фан-аутах.
Ведётся под ТЕКУЩИЙ стек делегирования; правила и приоритеты — в `CLAUDE.md`, раздел «Делегирование
субагентам».

## Текущий стек каналов

| Канал | Роль | Сильные стороны |
|-------|------|-----------------|
| **Kiro CLI** (MCP `kiro-cli`, первый приоритет) | нативный Kiro-агент: правит файлы в `workdir`, сессия на диске (`delegate`/`reply`), max reasoning | объёмные правки кода, второе мнение по дизайну; модели `claude-opus-4.8` → `claude-sonnet-4.6` → `glm-5` |
| **Codex** (MCP, второй приоритет) | глубокий анализ/ресёрч, чтение реального репозитория | внешне-API-фезибилити, «посчитай по реальному коду», адверсариальная критика |
| **Claude-Agent / Workflow** | свои Task/Agent-субагенты, структурный fan-out | фронт (FSD-слайсы), самопроверка eslint/vue-tsc, structured output для синтеза |
| **Главная модель** | оркестратор | сводит контракты, верифицирует чужие diff, держит когерентность по `CLAUDE.md` |

Per-account spacing Kiro: на одном Kiro-аккаунте вызовы сериализуются — реального параллелизма по
одному аккаунту нет; конкуренцию разводить по РАЗНЫМ каналам/директориям.

## Durable-уроки (channel-agnostic, из сборки блока «Автоматизация»)

1. **Замораживать ТОЧНЫЕ контракты ДО fan-out** — не только имена ресурсов, а точные пути/методы/формат
   ответа. Рассинхрон путей (`/automation/tasks/{id}/parse` vs `/automation/{id}/parse-targets`) и
   формата ответа (`data:{task,parse_run}` vs плоско) пришлось сводить руками, и они всплыли как живые
   500-е. Pre-flight «shared decisions» одним коротким решением срезает половину работы по сведению.
2. **Обязательная фаза Critique.** Адверсариальный критик поймал блокеры, которые не увидел никто:
   `auth()->id()` внутри клиента (в очереди `null→0`, ломает ownership/лимиты), отсутствие
   scheduler-контейнера, конфликт драйвера очереди. Без неё — красивый, но нерабочий дизайн.
3. **Партиционировать fan-out по непересекающимся файлам/каналам/директориям.** Два агента на одну зону
   расходятся по несущим решениям → дорого сводить. Либо один агент на зону, либо перекрытие делать
   ЯВНО адверсариальным (один проектирует, второй атакует). Конкурентная правка общего дерева работает
   без worktree, если скоупы заданы непересекающимися явно.
4. **Маршрутизация по сильным сторонам каналов** (подтверждено): внешне-API-фезибилити + «посчитай по
   коду» → Codex; адверсариальная критика архитектуры → in-house критик; независимое второе мнение по
   дизайну / объёмная правка → Kiro; фронт + самопроверка → Claude-Agent.
5. **Живая проверка ловит то, что типы/тесты не ловят.** Реальный прогон вскрыл 4 бага, невидимых для
   тестов: рассинхрон формата ответа store↔бэк, broadcast «payload too large» роняет job, реактивный
   дедлок `accountId`, авто-refresh review после async-парсинга. Гонять вживую, не только зелёные гейты.
6. **Агенты выходят за объявленный скоуп** (правят routes/provider, которые просили не трогать) И
   пропускают часть. Always verify: что сделано лишнего и что НЕ сделано. Швы (роуты, каналы, schedule,
   биндинги) сводит оркестратор последовательно после слияния слайсов.
7. **Инфра-грабли:** `queue:work` кэширует код при старте — после правок рестартить ВСЕ воркеры
   (queue-worker + automation-worker), иначе job падает на старом коде.
8. **Не предполагать «исправлено/разблокировано» без теста.** 2 неудачи подряд = не транзиент, не
   долбить. И сверять с фактическим `now()` ДО вывода о TZ-баге (длинная сессия — реальное время ушло).

## Что улучшить в следующей оркестрации

1. **Pre-flight «shared decisions» ДО fan-out** — канонические имена/пути/контракты одним решением.
2. **Острее партиционировать** — не два независимых дизайна одной зоны, а явная адверсариальность.
3. **Сохранять фазу Critique** — окупается (ловит блокеры).
4. **Structured output (schema) в Workflow** — упрощает синтез, дефолт для сводимого fan-out.
5. **Заземление перед fan-out** (чтения карты слоёв в main-контексте) — промпты ссылаются на реальные
   файлы/классы, агенты не плутают.
6. **Верификация делегатов — обязательна:** каждый diff/итог проверять самому (grep/тесты/чтение),
   черновики чужих моделей не принимать на веру.
